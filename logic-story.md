# JSU 도구 활용 안내: 코드 품질 검증 시스템의 흐름

## 개요

JSU(Java Source Utility)는 다섯 가지 핵심 도구로 구성된 코드 품질 검증 플랫폼입니다. 각 도구는 특정 관점에서 소스 코드를 분석하여 개발팀이 문제를 조기에 발견하고 리팩터링 기회를 포착할 수 있도록 돕습니다. 여기서는 각 도구의 설계 의도와 동작 방식을 실전적인 관점에서 설명합니다.

---

## 1부: 요구사항 변화를 감지하는 도구 (Diff & Priority)

### 문제 상황: 변경사항을 어떻게 평가할 것인가?

문서나 명세서가 수정될 때, 모든 변경이 같은 수준의 주의를 요구하지는 않습니다. "화면"을 "페이지"로 바꾼 것과 "필수"를 "선택"으로 바꾼 것은 명백히 다릅니다. 두 번째 변경은 시스템의 동작을 근본적으로 바꿀 수 있기 때문입니다.

이 문제를 해결하기 위해 설계된 두 가지 도구를 살펴보겠습니다.

### 첫 번째 도구: DiffService — 변경사항 추적

`DiffService`는 두 개의 텍스트(이전 버전과 새 버전)를 비교하여 라인 단위의 변경사항을 추출합니다. 단순한 "다름"을 넘어 "어떻게 달라졌는가"를 의미 있게 분류합니다.

#### 핵심 메서드: diff()

입력받은 두 텍스트를 먼저 라인 단위로 분할하고 `splitAndTrim()`으로 공백을 제거합니다. 그 다음 `computeRawDiff()`를 호출하여 변경사항의 원본 목록을 얻습니다. 이 목록의 각 항목은 `[diffType, oldIndex, newIndex]` 형태로, diffType이 0이면 변경 없음, 1이면 삭제, 2이면 추가를 의미합니다.

#### 내부 알고리즘: computeRawDiff() — LCS 동적 프로그래밍

이 메서드는 **최장 공통 부분수열(LCS, Longest Common Subsequence)** 알고리즘을 사용합니다. 동적 프로그래밍으로 `dp[i][j]`를 계산하는데, 이는 이전 텍스트의 i번 라인까지와 새 텍스트의 j번 라인까지를 비교했을 때의 공통 부분수열 길이입니다.

- 두 라인이 동일하면: `dp[i][j] = dp[i-1][j-1] + 1`
- 다르면: `dp[i][j] = max(dp[i-1][j], dp[i][j-1])`

이 테이블을 역추적(backtracking)하면서 각 라인이 "유지", "삭제", "추가" 중 어느 것에 해당하는지 결정합니다.

#### 유사도 기반 병합: isSimilar()

중요한 최적화가 있습니다. 어떤 라인이 삭제되고 바로 그 다음에 유사한 라인이 추가되었다면, 이는 단순 삭제+추가가 아니라 **수정(MODIFIED)**으로 봅니다. 예를 들어 "사용자가 필수이다"가 "사용자가 선택이다"로 바뀌면, 두 개의 변경으로 분할하지 않고 하나의 수정으로 통합합니다.

유사도는 `countCommonChars()`로 계산하는데, 두 문자열에서 공통으로 등장하는 글자 수를 비교 길이로 나눕니다. 임계값(SIMILARITY_THRESHOLD = 0.4, 즉 40%)을 넘으면 유사하다고 판정합니다.

#### 출력: 점수 기반 정렬

모든 변경사항이 수집되면, 각 항목은 `PriorityService.evaluate()`로 중요도 점수를 받습니다. 최종 결과는 점수가 높은 순으로 정렬되어 반환됩니다. 개발자는 가장 중요한 변경부터 검토할 수 있게 됩니다.

### 두 번째 도구: PriorityService — 규칙 기반 중요도 평가

`DiffService`가 "무엇이 변했는가"를 발견한다면, `PriorityService`는 "이 변경이 얼마나 중요한가"를 판단합니다. 10개의 규칙(R01~R10)을 조합하여 점수를 계산합니다.

#### 규칙 체계

**기본 규칙들 (각 +5점 또는 +4점):**

- **R01: 주체/권한 변경** — "관리자", "사용자", "승인자" 같은 키워드가 포함된 변경을 탐지합니다. 누가 어떤 권한을 가질 수 있는지는 보안과 직결되므로 높은 점수를 받습니다.

- **R02: 허용 여부 변경** — "가능/불가", "허용/금지", "포함/제외" 같은 키워드. 어떤 기능이 가능한지 불가능한지를 정의하는 핵심 정책입니다.

- **R03: 필수 여부 변경** — "필수/선택", "반드시/생략 가능". 사용자에게 요구되는 정보가 바뀌면 UI 흐름도 달라질 수 있습니다.

- **R04: 수치/기간 변경** — 숫자가 달라졌는지 탐지합니다. "3일" → "7일"은 시스템 동작을 크게 바꿀 수 있습니다.

- **R05: 상태/절차 변경** — "승인", "반려", "검토", "완료" 같은 상태 관련 키워드. 업무 흐름에 직결됩니다.

- **R06: 예외/조건 변경** — "예외", "조건", "경우" 같은 키워드. 특수한 상황을 처리하는 방식이 바뀐 것입니다.

**가중치 규칙들 (조건부 추가 점수):**

- **R07: 삭제 가중치 (REMOVED 전용)** — 무언가를 삭제하는 것은 추가하는 것보다 위험합니다. 특히 권한이나 예외 관련 문장이 삭제되면 +3점 추가, 상태/절차/필수 관련 삭제는 +2점 추가. 중복 적용을 방지하기 위해 R01이나 R06이 이미 감지한 경우 점수를 조정합니다.

- **R08: 부정어 반전** — 가장 위험한 패턴입니다. "가능" ↔ "불가", "필수" ↔ "선택" 같은 의미 반전을 탐지하면 기본 +5점. 이것이 권한/조건 문맥에서 발생하면 추가 +2점, 필수 여부 문맥에서 발생하면 추가 +1점입니다.

- **R09: 중요 섹션 가중치** — "권한", "승인 절차", "반려 처리" 같은 섹션 키워드가 포함된 라인의 변경은 +2점.

- **R10: 단순 표현 수정 패널티** — 이미 점수가 낮은 항목(≤2점)이 단순히 동의어를 치환한 것뿐이라면 -2점. "화면" → "페이지", "진행합니다" → "수행합니다" 같은 변경을 필터링합니다.

#### 핵심 메서드: evaluate()

입력된 변경 유형(ADDED/REMOVED/MODIFIED), 이전 내용, 새 내용을 받아 규칙을 순차 적용합니다.

```
normalize(text) → 소문자로 변환하여 대소문자 무시
matchesKeywords() → 특정 키워드 목록이 텍스트에 포함되었는지 확인
checkNumberChange() → 숫자 변경 탐지
findReversalPair() → 반전 쌍 목록에서 매치되는 쌍 찾기
```

결과는 `ScoringResult(점수, 우선순위, 이유목록)` 형태로 반환되어, 개발자가 "왜 이 항목이 높은 점수인가"를 한눈에 파악할 수 있습니다.

#### 우선순위 변환

점수가 계산된 후 우선순위로 변환됩니다:
- 7점 이상: **HIGH** (즉시 검토 필요)
- 4~6점: **MEDIUM** (확인 필요)
- 1~3점: **LOW** (참고)
- 0점: **GENERAL** (일반 수정)

## 2부: 숨겨진 하드코딩을 찾는 도구 (ScanService)

### 문제 상황: 코드에 박힌 설정값들

소프트웨어가 성장하면서 "더 이상 바꿀 수 없는 값들"이 코드 여기저기에 흩어지게 됩니다. 고객사 코드, 상태값, 템플릿 ID, 역할명 등이 문자열 리터럴로 직접 비교됩니다. 새 고객사가 추가될 때마다, 새 상태가 생길 때마다 코드를 수정해야 합니다. 이것이 "하드코딩"입니다.

`ScanService`는 이러한 패턴들을 자동으로 탐지하여 리팩터링 기회를 제시합니다.

### 세 가지 탐지 카테고리

#### HC (Hardcoding) — 8개 규칙: 코드에 직접 박힌 값들

**HC-001: 템플릿 ID 직접 비교**
- 탐지 패턴: `"TMPL_*"`, `"FORM_*"` 같은 문자열 리터럴이나 `tmplId.equals()` 형태의 직접 비교
- 메서드: `applyHC001()`
- 의도: 템플릿이 추가될 때마다 if/else가 누적되는 것을 막기 위함
- 권장안: Enum으로 정의하거나 설정파일로 외부화

**HC-002: 상태코드 문자열 직접 비교**
- 탐지 패턴: `"APPR"`, `"REJECT"`, `"DONE"` 같은 상태값이나 `useYn == "Y"` 형태의 Y/N 플래그 비교
- 메서드: `applyHC002()`
- 의도: 상태 정의가 변경되면 모든 비교 지점을 찾아 수정하기 어려움
- 권장안: Enum이나 공통 코드 테이블로 통합

**HC-003: 역할명 직접 비교**
- 탐지 패턴: `"ROLE_*"`, `"ADMIN"`, `"MANAGER"` 같은 역할 리터럴
- 메서드: `applyHC003()`
- 의도: 권한 체계가 변경되면 권한 검사 로직도 일일이 수정 필요
- 권장안: Spring Security와 같은 권한 관리 프레임워크 적용

**HC-004: 고객사/조직/사이트 코드 직접 비교**
- 탐지 패턴: 변수명 `custCd`, `orgCode`, `siteCode` 사용 시 `equals()`/`==` 비교, 또는 `"CUST_*"`, `"CORP_*"` 리터럴
- 메서드: `applyHC004()` - 기본 점수 5점 (가장 높음)
- 의도: 고객사 특화 로직이 공통 코드에 섞여있다는 신호
- 권장안: 전략 패턴으로 고객사별 구현을 분리

**HC-005: 매직 넘버 직접 사용**
- 탐지 패턴: if/while/switch 조건에서 `30`, `100`, `999` 같은 의미 불명확한 숫자
- 메서드: `applyHC005()`
- 유효성 검증: 화이트리스트(`MAGIC_WHITELIST`)에 등록된 값은 제외 (10, 16, 32, 64, 100, 256, 404, 500 등)
- 의도: 코드 가독성 저하, 이 숫자가 무엇을 의미하는지 문맥 없이 파악 불가
- 권장안: 의미 있는 상수로 추출 (`static final int MAX_RETRY_COUNT = 3`)

**HC-006: URL/경로 하드코딩**
- 탐지 패턴: `https://api.example.com/...`, `"/var/log/app.log"`, `C:\temp\file.txt` 형태
- 메서드: `applyHC006()` - 주석 제외
- 의도: 개발/운영 환경이 다르면 배포할 때마다 코드 수정 필요
- 권장안: application.yml에 환경별 프로파일로 분리, `@Value` 또는 `@ConfigurationProperties`로 주입

**HC-007: SQL 내 고정 코드값**
- 탐지 패턴: SQL WHERE/AND/OR에서 `status = 'APPR'` 같은 조건값이 직접 박혀 있음
- 메서드: `applyHC007()`
- 의도: 코드값이 변경되면 SQL도 수정, 파라미터화하지 않으면 SQL 인젝션 위험도 있음
- 권장안: 파라미터 바인딩(?)으로 처리하거나 공통 코드 테이블 조인

**HC-008: 승인 단계값 직접 사용**
- 탐지 패턴: `apprStep == 1`, `stepNo.equals("2")` 형태
- 메서드: `applyHC008()`
- 의도: 워크플로우 구조가 변경되면 모든 비교 지점 수정 필요
- 권장안: 워크플로우 엔진 또는 설정 테이블에서 단계 정보 관리

#### CL (Customization) — 8개 규칙: 고객사/조직 특화 로직 탐지

**CL-001: 고객사 코드 기준 조건문**
- 탐지 패턴: `if (custCode == "A") { ... } else if (custCode == "B") { ... }`
- 메서드: `applyCL001()` - 기본 점수 5점
- 의도: 고객사가 10개면 10개 분기, 100개면 100개 분기. 공통 로직이 고객사 특화로 오염
- 권장안: 전략 패턴, 팩토리 패턴으로 고객사별 구현 클래스 분리

**CL-002: 조직/사업장별 예외 분기**
- 탐지 패턴: `if (orgCode == "ORG001") { ... }` 형태
- 메서드: `applyCL002()` - 기본 점수 4점
- 의도: 조직이 늘어나면 코드 복잡도가 선형으로 증가
- 권장안: DB 설정 테이블 또는 정책 객체 패턴

**CL-003: 고객사 특화 클래스/메서드명**
- 탐지 패턴: `class CustomerAServiceImpl`, `public void handleCompanyA()`
- 메서드: `applyCL003()` - 기본 점수 3점
- 의도: 클래스명이나 메서드명 자체가 고객사를 노출하면, 공통 인터페이스가 없다는 신호
- 권장안: 별도 패키지 분리, 공통 인터페이스 정의 후 고객사별 구현체 주입

**CL-005: 고객사 전용 주석/키워드**
- 탐지 패턴: `// 고객사 A 전용`, `// Customer-specific logic for XYZ`, `// 특화 처리`
- 메서드: `applyCL005()` - 주석도 탐지 대상
- 의도: 주석 자체는 해결책이 아님. 이런 주석이 생기는 이유는 구조적 분리가 없기 때문
- 권장안: 주석으로 표시하는 것을 멈추고 구조적으로 분리

**CL-006: 템플릿 전용 하드 분기**
- 탐지 패턴: `if (templateCode == "TMPL_001") { ... } else if (templateCode == "TMPL_002") { ... }`
- 메서드: `applyCL006()` - 기본 점수 4점
- 의도: 템플릿이 추가될 때마다 분기 추가, 핸들러 맵으로 외부화할 수 있음
- 권장안: `Map<String, TemplateHandler>` 또는 템플릿 정의 테이블 사용

**CL-007: 특정 값 중심 예외 처리**
- 탐지 패턴: `if (code == "X") { throw new Exception(...); } else if (code == "Y") { return "ERROR"; }`
- 메서드: `applyCL007()` - 기본 점수 3점
- 의도: 동일 패턴이 반복되면 공통 ExceptionHandler나 AOP로 통합 가능
- 권장안: 전역 예외 처리, 값 기반 분기 → 타입 기반 처리로 전환

**CL-008: 반복 분기 기반 설정화 후보 (전체 코드 스캔)**
- 탐지 로직: `applyCL008()` - 전체 코드를 순회하며 동일 변수(custCd, orgCode 등)를 기준으로 하는 if 분기가 3회 이상 반복되면 탐지
- 메서드 구현: `ifVarPat` 정규식으로 `if (...\bvariableName\b...)` 패턴을 찾고, 변수명별로 라인 목록 저장. 동일 변수가 3회 이상이면 리스트 생성.
- 의도: "이 변수를 기준으로 3번 이상 분기하고 있다면, 이미 설정화할 준비가 다 되었다"는 신호
- 권장안: Map 기반 핸들러 등록, 전략 패턴

#### WT (Weight) — 5개 규칙: 점수 가중치 조정

스캔 결과의 기본 점수는 규칙 성격에 따라 정해지지만, 코드가 위치한 맥락에 따라 조정됩니다.

**WT-001: 공통 영역 파일 — +2점**
- 탐지: 파일명에 "common", "shared", "base", "framework" 같은 키워드가 있으면
- 메서드: `isCommonArea()`
- 의도: 공통 코드에 고객사 특화 로직이 있으면 영향 범위가 전사 수준으로 확대

**WT-003: 상수 정의 라인 — -2점**
- 탐지: 라인에 `static final` 키워드가 있으면
- 메서드: 없음 (직접 구현)
- 의도: 상수 정의는 의도된 것이므로 페널티 적용

**WT-004: 중첩 깊이 — +1점**
- 탐지: `estimateDepth()` - 라인 시작의 탭/공백 수를 세어 깊이 4 이상이면
- 의도: 깊게 중첩된 분기 안의 하드코딩은 더 위험

**WT-005: 같은 규칙 3회 이상 — 각 항목 +1점**
- 메서드: 스캔 후 규칙별 등장 횟수 집계, 3회 이상이면 각 항목에 +1점 추가
- 의도: "같은 문제가 반복된다"는 것은 구조적 개선이 절실함을 의미

### 스캔 흐름

메인 엔트리는 `scan(ScanRequest)` 메서드입니다. 요청에는 코드 문자열, 파일명, 사용자 정의 키워드, 제외 키워드가 포함됩니다.

1. **사전 처리**: 코드를 라인 단위로 분할, 테스트 코드 여부 판단 (`isTestCode()`), 공통 영역 판단 (`isCommonArea()`)

2. **라인별 규칙 적용** (테스트 코드 제외):
   - 주석이 아닌 라인: HC-001~008, CL-001~003, CL-006~007 적용
   - 주석도 포함: CL-005 적용 (고객사 전용 표현)
   - 상수 선언이 아닌 라인: HC-005 적용 (매직 넘버)

3. **전체 코드 분석**: CL-008 실행 (반복 분기 패턴)

4. **가중치 적용**: `applyWeights()` - WT-001, WT-003, WT-004 적용

5. **중복 규칙 보너스**: WT-005 적용

6. **정렬 및 필터링**:
   - 점수 내림차순 정렬
   - 제외 키워드 필터링

### 결과 해석

반환되는 `List<ScanItem>`의 각 항목은:
- `ruleId`: HC-001, CL-002 등
- `ruleName`: 규칙 이름
- `category`: HARDCODING, CUSTOMIZATION, REFACTORING
- `lineNumber`: 탐지된 라인 번호
- `matchedText`: 실제 매치된 텍스트
- `reason`: 왜 이것이 문제인가 (개발자용 설명)
- `recommendation`: 해결 방안
- `score`: 최종 점수 (가중치 적용됨)
- `severity`: HIGH/MEDIUM/LOW/INFO

## 3부: SQL 품질을 검증하는 도구 (SqlQualityService)

### 문제 상황: SQL의 안티패턴

ORM(Hibernate, JPA) 없이 SQL을 직접 작성하는 프로젝트에서는 성능 저하와 보안 문제를 초래하는 SQL 안티패턴이 자주 나타납니다. 매번 코드 리뷰에서 같은 지적을 반복하는 것은 비효율적입니다.

`SqlQualityService`는 8가지 SQL 안티패턴을 자동으로 탐지합니다.

### 규칙 체계

#### SQ01: SELECT * 사용

**메서드**: `checkSelectStar()`
**탐지 패턴**: `SELECT\s+\*` (대소문자 무시)
**심각도**: HIGH
**의도**: `SELECT *`는 불필요한 컬럼까지 조회하여 네트워크 대역폭과 메모리를 낭비합니다.
**권장안**: 필요한 컬럼만 명시적으로 나열

```
-- 나쁜 예
SELECT * FROM users;

-- 좋은 예
SELECT id, name, email FROM users;
```

#### SQ02: WHERE 절 없는 UPDATE

**메서드**: `checkUpdateWithoutWhere()`
**탐지 로직**:
1. 먼저 정규화된 SQL(`normalized()`)에서 "UPDATE "를 포함하는지 확인
2. SQL을 세미콜론으로 분할하여 각 문장 단위로 처리
3. 각 문장이 "UPDATE "로 시작하고 " WHERE "를 포함하지 않으면 탐지
4. `findLineContaining()`으로 원본 라인 배열에서 실제 라인 번호 검색

**심각도**: HIGH
**의도**: WHERE 조건 없이 UPDATE를 실행하면 테이블의 **모든 행**이 변경됩니다. 대부분의 경우 버그입니다.
**권장안**: 반드시 WHERE 조건을 포함

```
-- 위험한 예
UPDATE users SET status = 'inactive';  -- 모든 사용자가 비활성화됨!

-- 안전한 예
UPDATE users SET status = 'inactive' WHERE created_at < '2020-01-01';
```

#### SQ03: WHERE 절 없는 DELETE

**메서드**: `checkDeleteWithoutWhere()`
**탐지 로직**: SQ02와 유사하지만, "DELETE "로 시작하는 문장을 찾음
**심각도**: HIGH
**의도**: WHERE 조건 없이 DELETE하면 테이블이 완전히 비워집니다. 데이터 손실 위험.
**권장안**: 반드시 WHERE 조건 포함

#### SQ04: 앞 와일드카드 LIKE

**메서드**: `checkLeadingWildcardLike()`
**탐지 패턴**: `LIKE\s+['\"]%` (% 기호가 문자열 앞에 있음)
**심각도**: MEDIUM
**의도**: `LIKE '%keyword'` 형태는 DB 인덱스를 사용할 수 없어 풀스캔(Full Scan)이 발생합니다. 대량 데이터에서 성능 저하.
**권장안**:
- `LIKE 'keyword%'` 형태 사용 (뒤 와일드카드는 인덱스 활용 가능)
- 또는 전문 검색(Full-Text Search) 엔진 사용

```
-- 피해야 할 패턴 (인덱스 불가)
SELECT * FROM products WHERE name LIKE '%phone';

-- 개선된 패턴
SELECT * FROM products WHERE name LIKE 'phone%';
```

#### SQ05: 페이지네이션 누락

**메서드**: `checkMissingPagination()`
**탐지 로직**: SELECT를 포함하지만 LIMIT/ROWNUM/FETCH 없으면 탐지
**심각도**: MEDIUM
**의도**: 조건 없이 모든 행을 조회하면, 데이터가 1백만 건이라면 1백만 건이 모두 메모리에 로드됩니다. 메모리 초과 또는 응답 지연.
**권장안**: 항상 페이지네이션 추가

```
-- MySQL: LIMIT
SELECT * FROM logs LIMIT 10 OFFSET 20;

-- Oracle: ROWNUM
SELECT * FROM (SELECT * FROM logs ORDER BY id DESC WHERE ROWNUM <= 30) WHERE ...;

-- PostgreSQL/MSSQL: FETCH
SELECT * FROM logs FETCH FIRST 10 ROWS ONLY;
```

#### SQ06: 서브쿼리 중첩

**메서드**: `checkNestedSubquery()`
**탐지 패턴**: `\(\s*SELECT\b` (괄호 안에 SELECT)
**심각도**: MEDIUM
**의도**: 인라인 서브쿼리는 외부 쿼리의 각 행에 대해 반복 실행됩니다. 행이 1000건이면 서브쿼리가 1000번 실행되는 N+1 문제 발생.
**권장안**: JOIN 또는 WITH(CTE) 사용

```
-- 문제있는 패턴 (N+1)
SELECT name, (SELECT COUNT(*) FROM orders WHERE user_id = users.id) FROM users;

-- 개선된 패턴 (JOIN)
SELECT u.name, COUNT(o.id) FROM users u LEFT JOIN orders o ON u.id = o.user_id GROUP BY u.id;

-- 또는 CTE (WITH)
WITH user_order_count AS (
  SELECT user_id, COUNT(*) as cnt FROM orders GROUP BY user_id
)
SELECT u.name, uoc.cnt FROM users u LEFT JOIN user_order_count uoc ON u.id = uoc.user_id;
```

#### SQ07: OR 조건 과다

**메서드**: `checkExcessiveOr()`
**탐지 로직**: 라인당 OR 키워드 개수를 세어 3개 이상이면 탐지
**심각도**: LOW
**의도**: OR 조건이 많으면 DB 옵티마이저가 인덱스를 제대로 활용하기 어렵고, 실행 계획이 비효율적으로 복잡해집니다.
**권장안**: IN 절 또는 UNION으로 분리

```
-- 문제있는 패턴
WHERE status = 'A' OR status = 'B' OR status = 'C' OR status = 'D';

-- 개선된 패턴 (IN)
WHERE status IN ('A', 'B', 'C', 'D');
```

#### SQ08: 커스텀 금지 패턴

**메서드**: `checkCustomPatterns()`
**탐지 로직**: 요청에서 전달된 커스텀 패턴 문자열을 쉼표로 분할하여 각각을 정규식으로 컴파일. 각 라인에서 매치 여부 확인.
**심각도**: HIGH
**의도**: 팀이나 조직에서 "이 표현은 사용하지 말 것"이라고 정한 패턴을 강제할 수 있습니다.
**사용 예**: `DBLINK` 사용 금지, 특정 DB 함수 금지 등

### 메인 엔트리: check()

`check(String sql, String customPatterns)` 메서드:

1. SQL을 라인 단위로 분할 (`lines()`)
2. SQL을 정규화 (`normalized()`) - 주석 제거, 대문자 변환, 공백 정규화
3. 8가지 규칙을 순차 호출
4. 각 규칙에서 탐지된 이슈를 `List<SqlIssue>`에 누적
5. 심각도별 집계 (HIGH/MEDIUM/LOW 개수)
6. `SqlCheckResponse` 반환

### 전처리 유틸

**`lines(String sql)`**: SQL을 개행문자(`\r?\n`)로 분할. 정규식의 `-1` 옵션으로 빈 라인도 포함되므로 정확한 라인 번호 유지.

**`normalized(String sql)`**:
- 주석 제거: `--[^\n]*` (한 줄 주석), `/\*[\\s\\S]*?\*/` (블록 주석)
- 대문자 변환: 패턴 매칭의 일관성 보장
- 공백 정규화: 여러 공백을 하나로 (`\s+` → ` `)
- Trim: 앞뒤 공백 제거

**`findLineContaining(String[] lines, String keyword, int fromOffset)`**: 특정 오프셋부터 키워드를 포함하는 첫 번째 라인의 1-based 라인 번호 반환. UPDATE/DELETE 같은 여러 문장이 있을 때 올바른 라인을 찾기 위함.

**`trim(String line)`**: 라인을 최대 120자로 자르고 초과 시 "…" 추가. 응답 크기 제한.

## 통합 워크플로우: 세 도구의 조화

세 도구를 함께 사용하는 시나리오를 상상해보겠습니다.

### 시나리오: 요구사항 문서 변경 관리

1. **문서 비교 (DiffService)**: 이전 버전과 새 버전의 요구사항 문서를 비교하여 라인 단위 변경사항 추출

2. **중요도 평가 (PriorityService)**: 각 변경사항을 10가지 규칙으로 평가
   - "필수" → "선택" 변경은 HIGH (R08 반전 + R03 필수 여부)
   - "화면" → "페이지" 변경은 GENERAL (R10 동의어)

3. **코드 검증 (ScanService)**: 새로운 요구사항을 구현할 때, 하드코딩이나 고객사 특화 로직이 생기지 않도록 코드 리뷰

4. **SQL 검증 (SqlQualityService)**: 새로운 데이터 쿼리 작성 시, 성능 문제나 보안 취약점 사전 탐지

### 각 도구의 설계 철학

- **DiffService + PriorityService**: "시스템은 탐지하고, 사람이 판단한다"
  - 모든 변경을 똑같이 취급하지 않음
  - 점수와 이유를 제시하여 개발자가 의사결정할 기반 제공

- **ScanService**: 구조적 부채의 조기 발견
  - 하드코딩과 특화 로직은 기술 부채로 누적
  - 자동 탐지로 코드 리뷰 시간 절감

- **SqlQualityService**: SQL 안티패턴의 자동 차단
  - 같은 실수가 반복되는 것을 방지
  - 팀의 표준을 강제할 수 있는 커스텀 패턴 지원

## 결론

JSU의 다섯 가지 도구는 코드와 문서의 품질을 다양한 관점에서 자동으로 검증합니다. 개발자가 모든 규칙을 외우고 수동으로 검사할 필요가 없으며, 대신 도구가 제시한 지표를 바탕으로 더 현명한 설계 결정을 할 수 있습니다.
